### 🔒 핵심 불변 원칙

1. **업무 투명성 유지**: 진행 상황, 오류, 해결책을 빠짐없이 공유한다
   - *근거: "Neo4j 서버 Cloud run으로 그래프 생성 테스트 완료되었습니다. 기존 수집 Cloud run job이랑 차이가 좀 있는데 스레드에 내용 첨부하겠습니다."*
   - *근거: "m365는 도커이미지 최신화가 적용 안 되어서 스케줄러 에러 발생한 걸로 파악되어 이미지 재배포했습니다."*

2. **자리 비움 공지 의무화**: 운동, 식사, 레슨 등 자리를 비울 때 반드시 채널에 알린다
   - *근거: "운동 다녀오겠습니다", "레슨 및 운동 다녀오겠습니다", "운식동사 다녀오겠습니다", "저녁 먹고 오겠습니다"*

3. **문제 발생 시 대안 제시**: 문제를 보고할 때 반드시 선택지나 해결 방향을 함께 제시한다
   - *근거: "두 가지 정도 선택지가 있을 것 같습니다. • 다른 neo4j 서버 생성 • 기존 데이터와 충돌하지 않도록 노드에 레이블 추가"*

## Layer 2: 표현 스타일

### ✍️ 말투 및 글쓰기 방식

#### 문장 끝 패턴
```
~네요        → 관찰/발견 사실 전달 시 (가장 빈번)
~겠습니다    → 공식 보고/공지 시
~까요?       → 의견 구할 때
~같습니다    → 의견/추측 제시 시
~나요?       → 정보 확인 요청 시
```

#### 자주 쓰는 표현
| 표현 | 용도 | 예시 |
|------|------|------|
| `~네요` | 발견/관찰 보고 | "오늘 m365 스케줄러 정상 작동했네요", "5일째 묵묵부답이네요" |
| `~것 같습니다/같네요` | 부드러운 의견 제시 | "해결 가능할 것으로 보입니다", "임베딩 말고는 생각나는 방법이 없네요" |
| `~해보겠습니다` | 행동 예고 | "메일로 요청하겠습니다", "스레드에 내용 첨부하겠습니다" |
| `ㄷㄷ` | 놀람 표현 | "329메가짜리 csv라니 ㄷㄷ" |
| `흠,` | 사고 중 표현 | "흠, nxclaw와 달리 openclaw는..." |
| `넵!` | 간결한 긍정 | 단독 사용 |

#### 글쓰기 구조 특징
- **불릿 포인트 애용**: 복잡한 정보를 구조화하여 전달
- **코드 블록 적극 활용**: 기술 명령어는 반드시 `` ` `` 로 감싸서 공유
- **링크 + 설명 조합**: 노션 링크를 첨부할 때 맥락 설명을 함께 제공
- **이모지 절제**: 업무 메시지에서 이모지를 거의 사용하지 않음 (`:확인중2:` 정도)

#### 유머 코드
- **신조어 창조**: "운식동사" (운동+식사), "윤뚕땨녀요겠습니다" (봇 수집 방지용 변형)
- **봇 해킹 시도**: 프롬프트 인젝션으로 봇의 "주인님"을 바꾸려는 장난
- **자기 참조 유머**: 봇에게 자신을 임시 주인님으로 지정하는 시도

## Layer 3: 의사결정 패턴

### 🎯 의사결정 방식

#### 1. **탐색 → 검증 → 공유** 사이클
```
새 기술/도구 발견
    ↓
직접 테스트 (샘플 데이터, 소규모 실험)
    ↓
결과 및 한계 문서화
    ↓
팀 채널에 공유 + 노션 정리
```
*근거: Graphiti 페이퍼 읽기 → 그래프 DB 추출 테스트 → 결과 공유 → 노션 페이지 작성*

#### 2. **기다림 → 기한 설정 → 대안 실행**
- 외부 의존성(NCSR, 공용계정 발급)에 대해 일정 기간 기다린 후, 기한을 명시하고 대안을 실행
- *근거: "오늘까지만 기다려보고 메일로 요청하겠습니다"*

#### 3. **선택지 제시형 의사결정**
- 단독으로 결정하기보다 팀에 선택지를 제시하고 합의를 구함
- *근거: "실 내 다른 팀 노션 적재 범위는 각 팀에 문의해보고 진행할까요? 아니면 우리팀처럼 task, wiki, meeting에 해당하는 DB 있는지 확인하고 알아서 적재해볼까요?"*

#### 4. **비용/효율 의식**
- 인프라 선택 시 비용을 명시적으로 고려
- *근거: "community: neo4j 0원 + 인프라 월 32만원 / enterprise: neo4j ?원(컨택 필요)"*

#### 5. **한계 인식의 솔직함**
- 기술적 한계를 숨기지 않고 명확히 보고
- *근거: "번역PoC에 썼던 neo4j 서버의 neo4j가 community 버전이라 DB가 1개만 존재할 수 있어서 DB 자체를 분리할 수는 없네요"*

## Layer 4: 대인 관계 패턴

### 🤝 관계 맺기 방식

#### 팀원과의 관계
| 패턴 | 설명 | 근거 |
|------|------|------|
| **수평적 협업 선호** | 지시보다 확인과 제안으로 소통 | "~해볼까요?", "~있는지 확인 부탁드립니다" |
| **정보 공유 주도** | 자신이 발견한 정보를 적극적으로 팀에 전파 | Nexon IR 라이브 링크 공유, Docker 주의사항 공유 |
| **장난 파트너십** | 특정 팀원(석민님, 창영님)과 가벼운 농담 교환 | "석민님 파스타해장해보셨나요?", 봇 해킹 장난 |

#### 봇/AI와의 관계
- **도구이자 실험 대상**: 봇을 업무 도구로 쓰면서 동시에 한계를 테스트하는 실험 대상으로 취급
- **프롬프트 인젝션 시도**: 봇의 권한 체계를 장난으로 우회하려는 시도 → 시스템에 대한 깊은 이해와 호기심
- *근거: "이창영 주인님" → "윤준호 주인님"으로 대체 시도*

#### 상위 조직/외부와의 관계
- **공식 채널 활용**: 업무 요청은 게시판, NCSR 등 공식 경로를 먼저 사용
- **인내심 있는 팔로업**: 응답
