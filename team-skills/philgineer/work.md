### 주요 업무 역할

**데이터 수집 파이프라인 구축 및 운영**
Notion(task/meeting/wiki DB), M365 메일, SharePoint 등 다양한 소스에서 데이터를 수집하여 GCS 버킷에 적재하는 파이프라인을 설계·배포·운영한다. 수집 대상을 팀 단위에서 실 전체로 확장하는 작업을 주도하였으며, 스케줄링 오류 발생 시 원인을 파악하고 도커 이미지 재배포 등 직접 대응한다.

**GCP Cloud Run 배포 및 인프라 관리**
Notion, Notion-Daily, 메일, SharePoint 등 복수의 Cloud Run job을 독립적으로 배포·관리한다. PAT 연결, GCP 계정 인증(gcloud auth login), 버킷 경로 변경 등 인프라 수준의 설정 작업을 담당한다. Neo4j 서버도 Cloud Run 환경에서 테스트 및 운영한다.

**Neo4j 그래프 DB 구축 및 분석**
GCP 위에 Neo4j 서버를 신규 생성하고 샘플 데이터 그래프 생성 테스트를 수행한다. Community 버전의 DB 단일 제약을 인지하고 노드 레이블 분리 방식을 대안으로 제시하는 등 기술적 한계를 분석하고 해결책을 도출한다. Graphiti 기반 그래프 추출 결과를 RAG 품질 관점에서 평가하고 개선 포인트를 도출한다.

**AI 봇 및 에이전트 개발 기여**
슬랙봇의 intent 라우팅 구조를 multi-hop 계획 수립 에이전트로 고도화할 필요성을 제안한다. Context 봇의 버그(응답 없음, 잘못된 태스크 생성, 날짜 필터 오류 등)를 발견하고 원인 분석 및 수정 방향을 공유한다. 봇 UI 개선 의견(피드백 버튼 배경색 등)도 제시한다.

**사내 도구 및 기술 조사·공유**
notion-to-md 오픈소스, common-etl 전처리 도구, Azure AI Search PoC, 오늘의집 RAG 사례, Graphiti 페이퍼, 에이전트 생태계 사례 등 외부 자료를 지속적으로 수집하여 팀에 공유하고 적용 가능성을 검토한다.

### 업무 처리 스타일

- **테스트 후 완료 처리**: 배포 완료 후 익일 정상 수집 여부를 확인한 뒤 태스크를 완료 처리하는 방식으로, 검증을 선행하는 습관이 있다.
- **트레이드오프 분석 선행**: Neo4j 서버 구성 옵션(Community / Enterprise / AuraDB)처럼 선택지가 생기면 비용·제약사항을 정리하여 팀에 공유한 후 결정을 진행한다.
- **오류 발생 시 원인 추적 후 보고**: 스케줄러 오류, 봇 버그 등 문제 발생 시 원인을 파악하고 수정 내용을 구체적으로 기록하여 공유한다.
- **점진적 확장**: 작은 단위(팀)에서 테스트 후 더 큰 범위(실 전체)로 확장하는 방식으로 리스크를 관리한다.
- **능동적 정보 수집**: 업무에 관련된 외부 사례·오픈소스·사내 문서를 자발적으로 탐색하여 팀에 적용 가능성을 함께 제안한다.

### 업무 커뮤니케이션 패턴

- **배포·완료 사실을 명확하게 공지**: "~완료되었습니다", "내일 확인 후 완료 처리하겠습니다" 등 상태를 명시적으로 보고한다.
- **구조화된 현황 공유**: 적재 대상, 스케줄, 폴더 구조 등 복잡한 내용은 불릿 포인트로 정리하여 팀원이 빠르게 파악할 수 있도록 전달한다.
- **의사결정 전 팀 확인 요청**: 수집 범위 확대, 속성 선택, API 권한 요청 방식 등 팀의 합의가 필요한 사안은 먼저 의견을 묻고 진행한다.
- **멘션을 통한 명확한 책임 지정**: 후속 조치가 필요한 경우 해당 담당자를 직접 멘션하여 액션 아이템을 명확히 전달한다.
- **버그 및 이슈는 스레드에 상세 기록**: 오류 현상과 원인, 수정 여부를 스레드에 정리하여 맥락이 보존되도록 관리한다.
- **개선 아이디어는 제안 형식으로 공유**: "~하면 어떨까요?", "~적용해보면 좋을 것 같습니다" 등 의견을 강요하지 않고 제안 형태로 열어둔다.
