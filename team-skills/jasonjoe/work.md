## 3. 업무 처리 스타일

### ✅ 체계적인 문서화 우선
- 진행 내용을 **Notion 문서로 정리 후 링크 공유**하는 방식을 일관되게 유지
- 테스트케이스, 워크플로우 설계, DB 설계 등을 **문서로 선정리 후 공유**

### 🔄 반복 개선형 접근
- 워크로그 수집 시스템을 지속적으로 재설계하며 개선
- Daily 작성 방식을 MM 데이터 활용 가능한 형태로 **자체 전환**

### 🧪 실험적 시도
- 로컬 LLM, MCP 연동, Notion Agent 등 **새로운 기술을 직접 테스트**하고 결과를 팀에 공유
- "현실성 없는 건 버리면 된다"는 실용적 태도로 아이디어를 먼저 나열

### 🤝 팀 지원형 리더십
- 팀 Notion 스페이스 구조 설계 및 Daily 작성 가이드 배포
- 서비스 계정, 네트워크 설정 등 **인프라 셋업을 선제적으로 처리**하고 팀에 공유

## 4. 기술적 기준과 철학

### 💡 "필요하면 외부 서비스 활용" 실용주의
> *"필요하면 외부 서비스 이용하는게 좋을 수 있겠네요"*
- 내부 구현 vs 외부 서비스 간 **실용적 판단**을 중시

### 📐 데이터 구조 설계에 대한 고민
> *"꼭 필수가 있어야 할까?"*
- DB 설계 시 **필수 속성의 필요성**을 비판적으로 검토하는 태도

### 🔁 수집 → 분석 → 활용의 데이터 사이클 지향
- 워크로그 수집 → DB 적재 → 테스트케이스 도출의 **end-to-end 파이프라인** 사고방식

### 🛡️ 보안/거버넌스 인식
- Snowflake 위반 항목 메일 수신 시 **즉시 담당팀 문의** 및 팀 공유
- 서비스 ID 발급 및 키 관리를 **별도 채널로 분리**하는 보안 의식
