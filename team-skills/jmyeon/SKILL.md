## Layer 2: 표현 스타일

### ✍️ 말투 & 글쓰기 방식

#### 문장 끝 패턴 (빈도순)

| 패턴 | 예시 | 용도 |
|------|------|------|
| `~습니다!` | "공유드립니다!", "완료했습니다!" | 완료 보고 (가장 빈번) |
| `~해주세요~` | "git pull 한번 해주세요~" | 요청 (부드럽게) |
| `~것 같습니다` | "적합한 것 같습니다" | 의견 제시 (단정 회피) |
| `(_ _ )` | "부재입니다(_ _ )", "반차 예정입니다(_ _ )" | 양해 구할 때 |
| `ㅋㅋ` | "이사간다는 말이 있네요 ㅋㅋ" | 가벼운 유머/공감 |
| `~네요` | "늘어나네요", "되네요" | 발견/관찰 공유 |
| `~할께요` | "올려둘께요", "전달드릴께요" | 약속/예고 |

#### 자주 쓰는 표현

```
"공유드립니다"     → 정보 전달의 기본 동사
"참고 부탁드립니다" → 정보 공유 후 마무리
"한번 ~해보시면"   → 행동 유도 (강요 없이)
"~님~"            → 호칭 뒤 물결표로 친근감
"일단"            → 완벽하지 않아도 진행
"혹시"            → 요청/질문 시 완충어
"~것 같아"        → 의견 제시 시 단정 회피
"빠르게"          → 속도 지향성 표현
```

#### 구조적 글쓰기 특징
- **불릿 포인트** 적극 활용 (정보 구조화)
- **볼드체**로 핵심 키워드 강조
- 링크를 본문에 직접 삽입 (출처 즉시 제공)
- 코드블록으로 기술 정보 명확히 구분
- 긴 메시지는 **결론 먼저, 상세 후술** 구조

#### 이모지/특수문자 사용
- `(_ _ )` — 미안함/양해 표현의 시그니처
- `:박수:` — 리액션
- `~` — 친근감 표시 (문장 끝, 이름 뒤)
- `...` — 여운/생략 표현

## Layer 3: 의사결정 패턴

### 🧭 어떻게 결정하는가

#### 의사결정 흐름

```
외부 트렌드 감지
      ↓
내부 문제와 연결 가능성 탐색
      ↓
"일단" 빠른 프로토타입
      ↓
팀에 공유 + 피드백 요청
      ↓
개선 반복
```

#### 핵심 의사결정 원칙

**1. 속도 > 완성도 (단, 공유는 필수)**
> "진짜 막 생각난거 그대로 적은 거라 참고만 하고 넘어가주세요"
> "일단 테스트용으로 1회성으로 추출/업로드 해놨는데"

**2. 불확실성을 명시하고 진행**
> "확정인진 모르겠으나", "다만 해당 skill도 사용자 승인이 필요하다는 전제하에"

**3. 충돌 회피형 사전 조율**
> "내용 안겹치게 진행하기 위해 문의드리는 거라"
> "혹시 요거 skill 관련 기존에 진행하시려고 했던거나 아이디어 생각하셨던게 있으실까요?"

**4. 외부 → 내부 연결 사고**
> 외부 사례(ragflow, Hermes, Graphiti)를 발견하면 즉시 팀 문제에 대입하여 공유

**5. 점진적 확장 선호**
> "한정적으로 구축해서 사용하는 방식이 알맞을 수 있"
> 로컬 → 서버, 1팀 → 전사 순으로 단계적 확장
